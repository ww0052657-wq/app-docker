The system,
one exercise at a time

You start with a jar and a laptop full of things you installed by hand. You finish with five containers, two databases, and a browser page that changes a service two hops away.

a running container a database a boundary published host port broken / absent

These are C4 container diagrams. In C4 a "container" means a separately runnable thing — an app, a database, a web server. It lines up almost exactly with a Docker container here, which is convenient but a coincidence.

1

Put it in a box

branch exercise-01 · ~65 min
What's new: fx-app-spring/Dockerfile (multi-stage) and .dockerignore. The app runs with no JDK, no Maven and no MySQL involved.
Watch for: the container starts and /health is green while /api/rates is a 500. "Up" and "working" are different claims — the pool is lazy. We deliberately do not fix the database here.
2

The whole stack in one file

branch exercise-02 · ~60 min
What's new: docker-compose.yml and .env.example at the workspace root. The database is part of the system now, not part of your laptop.
Watch for: down keeps the volume, down -v destroys it — and the seed only re-runs against an empty data directory. That one fact explains most "I changed the seed and nothing happened" bugs.
3

Liquibase owns the schema

branch exercise-03 · ~70 min
What's new: db/changelog/** inside the app; the init-script mount is deleted. One owner for the schema.
Watch for: the three breakages you cause on purpose — an edited applied change set (checksum), bad DDL (fails, not recorded), and a stuck DATABASECHANGELOGLOCK.
4

A second app, and the port fight

branch exercise-04 · ~45 min
What's new: fx-monitor/ as a sibling folder and a third service on host port 3000. Your API's build context did not grow by a byte.
Watch for: Bind for 0.0.0.0:8080 failed: port is already allocated. Container ports are private; published host ports are shared. It is always the left-hand number.
5

The orchestrator loop

branch exercise-05 · ~80 min
What's new: fx-orchestrator/ + orchestrator-db, your POST /api/feed/rates, an outbound ACK over RestTemplate, the ACCEPTING toggle, and change set 009 on a live database.
Watch for: your answer sets the upstream cadence — ACCEPTED means the next batch in 2s, DECLINED means 10s. You are on the deciding end of someone else's loop.

What's actually going on — the feed loop

Watch for: The ACK is a second HTTP call, not the response to the first — which is why your app has to be an HTTP client as well as a server. That is backpressure, and you control it.

What “done” looks like

end of Task 5
What's new: rates moving every two seconds, numbers flashing green or red, ~90s of history per pair, feed interval 2.0 s.
Watch for: flip ACCEPTING off and it all freezes — trend lines go flat, Last update climbs and turns amber, and the orchestrator drops to one batch every ten seconds.
