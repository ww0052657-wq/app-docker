Today's tasks

1 · Put it in a box

AM · ~65 min · deck: Containerisation / Dockerfiles

Why: "works on my machine" is a bug report, not a status.

Start a new repo, and no database needed — stop MySQL for this one. Lay the workspace out first (three siblings, README with your name), then write the Dockerfile in stages and run it after each: naive single-stage → publish a port → diagnose the whose localhost? failure (and deliberately not fix it) → multi-stage → prove the layer cache by touching a file → .dockerignore → override config with -e → volumes and bind mounts → the everyday ps/logs/exec/images commands.

Stuck? The container starting is not the app working — the pool is lazy, so /health is green while /api/rates is 500. Read the container log, not the HTTP body.
Checkpoint: /health UP · / 404 · /api/health/db DOWN · /api/rates 500 — running with no JDK, no Maven and no MySQL. Multi-stage image smaller; rebuild leaves go-offline CACHED. The 10 rates arrive in Task 2.

2 · The whole stack in one file

AM · ~60 min · deck: Containerisation

Why: the app was portable, the database still wasn't.

A db service and an app service, wired by service name not localhost. Healthcheck + depends_on: condition:, named volume vs bind mount, .env and ${VAR:-default}, and the down / down -v experiment that explains why your seed edits "did nothing".

Stuck? 3307:3306 — left is your laptop, right is inside. Container-to-container traffic uses the right one.
Checkpoint: local MySQL stopped, docker compose up serves 10 rates; down -v && up re-seeds from scratch.

3 · Liquibase owns the schema

PM · ~70 min · deck: Data migrations (gap-fill)

Why: the shape of your database is the last thing you own that isn't code.

Change sets in YAML, seed data as sqlFile, the init-script mount deleted, and DATABASECHANGELOG read as the ledger it is. Then break it three ways on purpose — an edited applied change set, bad DDL, and a stuck DATABASECHANGELOGLOCK — and fix each.

Stuck? Never edit a change set that has already run. The fix is always a new one. clearCheckSums means "trust the files, forget what actually ran".
Checkpoint: with no seed file mounted, down -v && up rebuilds schema + data: 8/20/30/200 rows, EUR/USD 1.0818.

4 · A second app, and the port fight

PM · ~45 min · deck: Containerisation

Why: most containers you'll ever run were built by someone else.

fx-monitor is given complete — containerise it, add it to the stack, hit port is already allocated, and work out which of the two numbers caused it. nginx serves the page and proxies /api onward, so the browser stays same-origin and CORS never enters the picture.

Stuck? Container ports are private to each container; published host ports are shared across your whole machine. It is always the left-hand number.
Checkpoint: three services up · http://localhost:3000 renders the live table · the toggle is greyed out with a 404 banner (correct — that's Task 5).

5 · The orchestrator loop

PM · ~80 min · deck: Containerisation + REST

Why: a service you don't control is now depending on your answer.

fx-orchestrator pushes a batch of rates every two seconds and waits for your ACK. Build the inbound endpoint (202), store the ticks, call out over RestTemplate with ACCEPTED or DECLINED, and add the toggle that decides which. Evolve fx_rate forward with a new change set — Task 3's rule, in production form.

Stuck? Read the orchestrator's log first — it names the exact URL, the exact JSON, and what it thinks you got wrong.
Checkpoint: toggle ON → values move, batches every 2s, fx_rate grows · OFF → frozen, DECLINED, 10s · /api/rates still returns exactly 10 rows either way.

Theory background

A container is a process, not a computer

No second operating system is booted. Your process runs on the same kernel as everything else, with its own view of the world: its own filesystem, its own process list, its own network interface. That is why it starts in milliseconds and why localhost means something different inside it.

Images are layers, and order is everything

Each instruction adds a read-only layer; a container adds one thin writable layer on top. Docker re-runs a layer only when the files it copies have changed — and every layer after it. Copy pom.xml and resolve dependencies before copying src, and a source edit rebuilds in seconds instead of minutes.

Two ports, two different jobs

-p 3000:80 — the left is a port on your laptop, a scarce machine-wide resource; the right is inside the container, private to it. Ten containers can all use port 80 internally. Only one can publish on host 3000. Every "port is already allocated" is about the left-hand number.

Compose networking: names, not addresses

Every service gets a DNS name equal to its service name, resolved on an internal network. fx-db:3306 works and doesn't care what IP the container got today. And "started" ≠ "ready": a healthcheck plus depends_on: condition: service_healthy is the difference between a reliable stack and one that works on the second try.

Migrations: the schema is code too

Liquibase keeps an ordered list of change sets and records in the database itself which ones it has run. Empty database → whole schema. Current database → nothing. Three behind → three applied. The one rule: never edit a change set that has already run. It was hashed when it ran; change the file and the hashes disagree, and Liquibase stops rather than guess.

Extra challenges

Shrink the image

Try eclipse-temurin:21-jre-alpine, then a jlink-trimmed runtime. Record the sizes. Then find the cost: what changed about libc, DNS, or your JVM flags? A smaller number that breaks in production is not a win.

Run as a non-root user

Add a USER to the runtime stage and make the app still work — including the log volume. This is the single highest-value hardening step for a container, and the one most likely to break something quietly.

Roll one back

Every change set you wrote has a rollback: block. Use the Liquibase Maven plugin to roll back the last one, look at what happened to DATABASECHANGELOG, and then reapply it. When is rollback genuinely usable, and when is "roll forward" the only honest option?

Make the feed misbehave

fx-orchestrator reads fx.delay.* and fx.ack.timeout from its environment. Set the ACK timeout below your own response time and watch it declare NO_ACK for batches you did answer. That's a real distributed-systems failure — the answer arrived, just too late to count.
